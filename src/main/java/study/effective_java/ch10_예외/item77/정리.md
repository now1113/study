:book: 아이템 77: 예외를 무시하지 말라

## 핵심
**빈 catch 블록(또는 예외를 삼켜버리는 처리)은 거의 항상 버그를 숨긴다.**  
예외는 "무언가가 예상과 달리 흘렀다"는 신호인데, 그 신호를 지워버리면 문제는  
**조용히 더 큰 문제로 성장한다.**

## 예외를 무시하면 생기는 문제
- **원인 추적 불가**: 장애가 나도 로그/스택트레이스가 없어 "왜"를 못찾음
- **불변식 붕괴**: 실패했는데 성공한 것처럼 다음 로직 진행 -> 데이터/상태가 꼬임
- **2차 장애 유발**: 원래 예외는 사소했는데, 이후 더 큰 예외(NPE/데이터 손상) 발생

## 예시: 아무것도 하지 않는 catch

```java
try {
    // something
} catch (SomeException e) {
    // do nothing
}
```

이 코드는 **예외를 무시한다는 의도조차 드러나지 않아서** 특히 더 위험하다.  
(나중에 읽는 사람은 "실수로 비워둔 건가?" "여기서 더 실패해도 되나"를 전혀 알 수 없음)

## 예외가 발생했을 때 선택지들

### 복구(recover)
- 대체값 사용, 재시도, 다른 경로로 진행 등
- 단, "정말로 안전하게" 복구 가능한지 판단 필요

### 전파(propagate)
- 호출자에게 던져서 상위에서 처리하게 함
- 흔히 "이 계층이 책임질 수 없는 실패"에 적합

### 변환(translate/wrap)
- 저수준 예외를 도메인/의미 있는 예외로 감싸 전달
- 예: `IOException` to `OrderExportException`

### 기록(log)
- 즉시 복구/전파가 어렵다면, 최소한 **원인과 맥락을 남긴다.**
- (로그는 "처리"가 아니라 "최소한의 책임"에 가깝고, 남발도 주의)

### 실패를 명확히 드러내기(fail fast)
- "이 상황이 오면 프로그램이 더 진행되면 안 된다"면 즉시 중단(예외 재던짐)

## 무시가 허용되는 드믄 경우
예외를 무시하는게 **완전히 틀린 건 아니지만**, 가능한 경우가 매우 제한적이다.

- 실패해도 프로그램의 정합성/안전성에 영향이 없고
- 실패가 기대 가능한 정상 흐름이며
- 실패 여부를 굳이 알 필요가 없거나, 다른 경로에서 이미 처리됨이 확실할 때

그리고 **이런 경우에도 그냥 비워두지 말고,** 의도를 드러내야 하낟.
- **주석으로 왜 무시해도 되는지** 남기기
- 예외 변수명을 `ignored`같은 이름으로 두어 의도를 표현하기

("무시"는 기술적 선택이라기 보다는 **명세/의도 표현의 문제**까지 포함한다)


## 결론
- **예외를 잡았다면, 그 catch는 반드시 의미 있는 행동을 해야 한다.**
- 복구/전파/변환/기록/실패 중 하나를 선택하고, "무시"는 정말 예외적인 상황에서만, 그 이유를 코드에 남기자
